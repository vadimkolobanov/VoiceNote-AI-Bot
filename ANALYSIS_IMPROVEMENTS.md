# Анализ кода бота: области для улучшения

## 🔴 Критические проблемы

### 1. **Отсутствие retry-логики для внешних API**
**Проблема:** При сбоях DeepSeek API или Yandex STT запросы не повторяются, что приводит к потере данных пользователя.

**Где:** 
- `src/services/llm.py` - функция `_call_deepseek_api()`
- `src/services/stt.py` - функция `recognize_speech_yandex()`

**Решение:**
- Добавить retry с экспоненциальной задержкой (например, библиотека `tenacity`)
- Обрабатывать специфичные ошибки (429, 503, таймауты)
- Логировать количество попыток

### 2. **Отсутствие connection pooling для HTTP-клиентов**
**Проблема:** Каждый запрос к внешним API создает новую `ClientSession`, что неэффективно.

**Где:**
- `src/services/llm.py:70` - создается новая сессия для каждого запроса
- `src/services/stt.py:25, 80` - аналогично

**Решение:**
- Создать глобальный `aiohttp.ClientSession` с пулом соединений
- Переиспользовать сессию для всех запросов
- Правильно закрывать при shutdown

### 3. **Потенциальная утечка паролей в логах**
**Проблема:** `DATABASE_URL` содержит пароль и может попасть в логи при ошибках.

**Где:**
- `src/core/config.py:33` - формирование `DATABASE_URL`
- Любые места, где логируется `DATABASE_URL` или ошибки подключения

**Решение:**
- Маскировать пароль в логах (заменять на `***`)
- Использовать отдельные переменные для логирования

### 4. **Отсутствие валидации длины текста**
**Проблема:** Нет проверки максимальной длины текста перед отправкой в LLM, что может привести к ошибкам или большим расходам.

**Где:**
- `src/bot/modules/notes/services.py:114` - функция `process_and_save_note()`
- `src/services/llm.py` - все функции обработки текста

**Решение:**
- Добавить проверку максимальной длины (например, 5000 символов)
- Обрезать или предупреждать пользователя при превышении

## 🟡 Важные улучшения

### 5. **Отсутствие circuit breaker**
**Проблема:** При частых сбоях внешних сервисов бот продолжает отправлять запросы, что усугубляет проблему.

**Решение:**
- Реализовать circuit breaker для DeepSeek и Yandex API
- Временно отключать функциональность при критических сбоях
- Уведомлять пользователей о временной недоступности

### 6. **Недостаточная обработка ошибок LLM**
**Проблема:** При ошибках парсинга JSON от LLM пользователь получает техническое сообщение.

**Где:**
- `src/services/llm.py:34-48` - функция `_parse_llm_json_response()`
- `src/bot/modules/notes/services.py:172` - обработка ошибок LLM

**Решение:**
- Улучшить парсинг (попытка извлечь JSON из markdown блоков)
- Fallback на базовую обработку при ошибках LLM
- Более дружелюбные сообщения пользователю

### 7. **Отсутствие rate limiting**
**Проблема:** Нет защиты от злоупотреблений и DDoS-атак.

**Где:**
- Веб-API (`src/web/api/`)
- Обработчики бота

**Решение:**
- Добавить rate limiting через Redis (например, `slowapi` для FastAPI)
- Разные лимиты для разных типов запросов
- Логирование подозрительной активности

### 8. **Глобальная переменная для пула БД**
**Проблема:** Использование глобальной переменной `db_pool` может привести к проблемам при тестировании и параллельном доступе.

**Где:**
- `src/database/connection.py:257` - `db_pool: asyncpg.Pool | None = None`

**Решение:**
- Использовать dependency injection
- Или создать singleton через класс с методами

### 9. **Отсутствие кэширования LLM-запросов**
**Проблема:** Одинаковые запросы к LLM обрабатываются повторно, увеличивая расходы и время ответа.

**Решение:**
- Кэшировать результаты `classify_intent()` для одинаковых текстов
- Использовать Redis для кэширования
- TTL кэша: 1-24 часа в зависимости от типа запроса

### 10. **N+1 запросы в некоторых местах**
**Проблема:** В циклах выполняются отдельные запросы к БД вместо batch-запросов.

**Где:**
- `src/services/scheduler.py:341` - `generate_and_send_daily_digest()` - может быть оптимизировано
- `src/services/scheduler.py:424` - отправка напоминаний о днях рождения

**Решение:**
- Использовать `asyncio.gather()` для параллельных запросов
- Batch-запросы где возможно
- Проверить другие места с циклами

## 🟢 Улучшения качества кода

### 11. **Недостаточная типизация**
**Проблема:** Многие функции не имеют полной типизации, что усложняет поддержку.

**Где:**
- Множество функций в репозиториях и сервисах

**Решение:**
- Добавить type hints везде
- Использовать `mypy` для проверки типов
- Добавить в CI/CD

### 12. **Дублирование кода**
**Проблема:** Похожая логика обработки ошибок повторяется в разных местах.

**Где:**
- Обработка ошибок Telegram API
- Форматирование сообщений об ошибках

**Решение:**
- Создать утилиты для обработки ошибок
- Единый формат сообщений об ошибках

### 13. **Отсутствие мониторинга и метрик**
**Проблема:** Нет метрик производительности, что затрудняет оптимизацию.

**Решение:**
- Добавить метрики (Prometheus или встроенные)
- Отслеживать:
  - Время ответа LLM
  - Количество ошибок по типам
  - Использование ресурсов БД
  - Активность пользователей

### 14. **Недостаточное тестирование**
**Проблема:** Очень мало тестов, особенно интеграционных.

**Где:**
- `tests/` - только несколько unit-тестов

**Решение:**
- Добавить интеграционные тесты для критических путей
- Тесты для обработки заметок
- Тесты для scheduler
- Mock внешних API

### 15. **Отсутствие валидации входных данных**
**Проблема:** Не все пользовательские данные валидируются перед обработкой.

**Где:**
- Текст заметок (может содержать вредоносный контент)
- Параметры API запросов

**Решение:**
- Использовать Pydantic для валидации
- Санитизация HTML в пользовательских сообщениях
- Проверка длины и формата

### 16. **Отсутствие обработки больших файлов**
**Проблема:** Нет ограничения на размер голосовых сообщений, что может привести к проблемам.

**Где:**
- `src/services/stt.py` - обработка аудио

**Решение:**
- Проверять размер файла перед загрузкой
- Ограничивать максимальную длительность (например, 5 минут)
- Предупреждать пользователя

### 17. **Неоптимальная работа с транзакциями**
**Проблема:** Некоторые операции могут быть атомарными, но выполняются отдельно.

**Где:**
- Создание заметки + начисление XP + проверка достижений

**Решение:**
- Использовать транзакции для связанных операций
- Обеспечить консистентность данных

### 18. **Отсутствие graceful shutdown**
**Проблема:** При остановке бота могут теряться данные в процессе обработки.

**Где:**
- `src/main.py:55` - `on_shutdown()`

**Решение:**
- Дождаться завершения всех фоновых задач
- Сохранить состояние FSM
- Закрыть все соединения корректно

### 19. **Недостаточная обработка таймаутов**
**Проблема:** Таймауты установлены, но не всегда обрабатываются корректно.

**Где:**
- Запросы к внешним API
- Запросы к БД

**Решение:**
- Единая стратегия таймаутов
- Retry с увеличенным таймаутом
- Логирование таймаутов

### 20. **Отсутствие health checks**
**Проблема:** Нет способа проверить состояние бота и зависимостей.

**Решение:**
- Добавить `/health` endpoint в FastAPI
- Проверять подключение к БД, Redis, внешним API
- Возвращать статус готовности
